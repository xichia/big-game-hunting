# Artifact Authority

This document defines how much authority each artifact type has.

## SOURCE OF TRUTH

Authoritative project state.

Examples:

- `GAME_TRUTH.md`
- `DESIGN_PRINCIPLES.md`
- `MECHANICS.md`
- `DECISION_LOG.md`

Rules:

- Requires human ratification.
- Must be paired with a decision log entry.
- Should not include speculative material.

## DRAFT ONLY

Exploratory material that may inform decisions but is not official.

Examples:

- broad game idea lists
- analogue research
- speculative mechanic sketches
- theme explorations

Rules:

- Must not be cited as project truth.
- Must not update ratified mechanics.

## PROBE

A small test artifact designed to validate or invalidate an assumption.

Examples:

- one-page ruleset
- simulated play transcript
- paper prototype
- test scoring table
- turn sequence

Rules:

- Tests only a bounded question.
- Does not establish truth on its own.

## REVIEW

An independent evaluation of an artifact.

Examples:

- feasibility review
- balance review
- simplicity/depth review
- source-of-truth conflict review

Rules:

- Advisory only.
- Must not ratify decisions.
- Should use GREEN / YELLOW / RED / QUESTION labels.

## PROPOSED SOURCE OF TRUTH UPDATE

A recommended update prepared for human approval.

Rules:

- Must specify exact files and exact changes.
- Does not become authoritative until ratified.

## RATIFIED

A human-approved decision.

Rules:

- Must be recorded in `DECISION_LOG.md`.
- May update source-of-truth files.

## VETOED

A blocked artifact, mechanic, or direction.

Reasons may include contradiction, risk, quality failure, or human decision.
